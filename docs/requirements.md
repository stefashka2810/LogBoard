# Требования LogBoard

---

# Управление пользователями

## Функциональные требования

| ID    | Название                    | Описание                                                                    | Приоритет |
|-------|-----------------------------|-----------------------------------------------------------------------------|-----------|
| FR-U1 | Регистрация пользователя    | Система должна позволять пользователю регистрироваться по username и паролю | HIGH      |
| FR-U2 | Аутентификация пользователя | Система должна обеспечивать вход пользователя в систему                     | HIGH      |
| FR-U3 | Выход из системы            | Система должна предоставлять пользователю возможность выхода                | MEDIUM    |
| FR-U4 | Изменение пароля            | Система должна предоставлять пользователю возможность изменять пароль       | LOW       |

## Нефункциональные требования

### Пароли и username

| ID     | Описание                                                                            | Приоритет |
|--------|-------------------------------------------------------------------------------------|-----------|
| NFR-U1 | Минимальная длина пароля пользователя должна составлять не менее 8 символов         | HIGH      |
| NFR-U2 | Пароль пользователя должен включать только допустимые символы                       | HIGH      |
| NFR-U3 | Пароли пользователей должны храниться в виде криптографического хэша                | HIGH      |
| NFR-U4 | При хэшировании паролей должна использоваться криптографическая соль                | HIGH      |
| NFR-U5 | Username пользователя должен включать только допустимые символы                     | HIGH      |
| NFR-U6 | Минимальная длина username пользователя должна составлять не менее 4 символов       | HIGH      |
| NFR-U7 | Система должна обеспечивать уникальность значений username среди всех пользователей | HIGH      |

### Аутентификация и токены

| ID      | Описание                                                                        | Приоритет |
|---------|---------------------------------------------------------------------------------|-----------|
| NFR-U8  | Система должна использовать access token для аутентификации запросов            | HIGH      |
| NFR-U9  | Access token должен иметь ограниченное время жизни                              | HIGH      |
| NFR-U10 | Система должна использовать refresh token для обновления access token           | HIGH      |
| NFR-U11 | Refresh token должен храниться и обрабатываться безопасным способом             | HIGH      |
| NFR-U12 | Система должна поддерживать обновление access token без повторного ввода пароля | HIGH      |

# Управление проектами

## Функциональные требования

| ID    | Название          | Описание                                                            | Приоритет |
|-------|-------------------|---------------------------------------------------------------------|-----------|
| FR-P1 | Создание проекта  | Система должна позволять пользователю создавать новый проект        | HIGH      |
| FR-P2 | Просмотр проектов | Система должна отображать проекты, в которых пользователь участвует | HIGH      |
| FR-P3 | Удаление проекта  | Система должна позволять удалять проект                             | LOW       |

## Нефункциональные требования

| ID     | Описание                                                                 | Приоритет |
|--------|--------------------------------------------------------------------------|-----------|
| NFR-P1 | Пользователь, создавший проект автоматически получает в нем роль `OWNER` | HIGH      |

# Управление участниками

## Функциональные требования

| ID    | Название                 | Описание                                                              | Приоритет |
|-------|--------------------------|-----------------------------------------------------------------------|-----------|
| FR-M1 | Добавление участника     | Система должна позволять добавлять пользователей в проект             | HIGH      |
| FR-M2 | Назначение ролей         | Система должна позволять назначать роль участнику проекта             | HIGH      |
| FR-M3 | Ограничение прав доступа | Система должна ограничивать доступ к функциям по роли                 | HIGH      |

## Нефункциональные требования

| ID     | Описание                                                                                                                             | Приоритет |
|--------|--------------------------------------------------------------------------------------------------------------------------------------|-----------|
| NFR-M1 | Система должна обеспечивать разграничение прав доступа к операциям управления участниками на основе ролей `OWNER`, `ADMIN`, `READER` | HIGH      |
| NFR-M3 | Система должна разрешать назначение и изменение ролей участников проекта только пользователю с ролью `OWNER`                         | HIGH      |
| NFR-M4 | В каждом проекте должен существовать ровно один участник с ролью `OWNER`                                                             | HIGH      |
| NFR-M5 | Пользователь с ролью `ADMIN` может добавлять и удалять участников проекта только с ролью `READER`                                    | HIGH      |
| NFR-M6 | Пользователь с ролью `OWNER` может добавлять и удалять участников проекта независимо от их роли                                      | HIGH      |

# Работа с ключами проекта

## Функциональные требования

| ID    | Название                    | Описание                                                                                              | Приоритет |
|-------|-----------------------------|-------------------------------------------------------------------------------------------------------|-----------|
| FR-K1 | Создание ключа проекта      | Система должна предоставлять возможность создания API-ключа для проекта                               | HIGH      |
| FR-K2 | Просмотр ключей проекта     | Система должна предоставлять список API-ключей проекта без отображения их полного значения            | HIGH      |
| FR-K3 | Отзыв ключа проекта         | Система должна предоставлять возможность отзыва (деактивации) API-ключа                               | HIGH      |
| FR-K4 | Использование ключа проекта | Система должна обеспечивать аутентификацию запросов отправки логов проекта с использованием API-ключа | HIGH      |

## Нефункциональные требования

| ID     | Описание                                                                                                              | Приоритет |
|--------|-----------------------------------------------------------------------------------------------------------------------|-----------|
| NFR-K1 | Управление API-ключами должно быть доступно только пользователям с ролями `OWNER` и `ADMIN`                           | HIGH      |
| NFR-K2 | API-ключи проекта должны храниться в виде криптографического хэша, полученного с использованием алгоритма HMAC-SHA256 | HIGH      |
| NFR-K3 | Полное значение API-ключа должно отображаться пользователю только один раз в момент создания                          | HIGH      |

# Работа с логами

## Функциональные требования

| ID    | Название             | Описание                                                                                                         | Приоритет |
|-------|----------------------|------------------------------------------------------------------------------------------------------------------|-----------|
| FR-L1 | Прием логов          | Система должна принимать логи от внешних приложений через API или чтение топика Kafka                            | HIGH      |
| FR-L2 | Просмотр логов       | Система должна предоставлять возможность просмотра логов проекта                                                 | HIGH      |
| FR-L3 | Фильтрация логов     | Система должна предоставлять возможность фильтрации логов по уровню лога и временному периоду                    | HIGH      |
| FR-L4 | Поиск по логам       | Система должна поддерживать поиск по содержимому логов независимо от регистра                                    | HIGH      |
| FR-L5 | Построение таймлайна | Система должна формировать таймлайн на основе агрегированных исходных данных с распределением по временной шкале | HIGH      |

## Нефункциональные требования

| ID     | Описание                                                                                  | Приоритет |
|--------|-------------------------------------------------------------------------------------------|-----------|
| NFR-L1 | Просмотр логов проекта должен быть доступен всем участникам проекта независимо от их роли | HIGH      |
| NFR-L2 | Система должна принимать логи только при наличии действительного API-ключа проекта        | HIGH      |
